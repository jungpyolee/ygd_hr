## 1. 핵심 라이팅 원칙

### 1-1. 5가지 라이팅 코어밸류 `[확인됨 - toss.tech]`

토스의 모든 제품 텍스트는 다음 5가지 코어밸류를 기반으로 작성된다.


| 코어밸류                 | 의미                   | 적용 기준                   |
| -------------------- | -------------------- | ----------------------- |
| **Clear (명확한)**      | 모호하지 않은 언어로 즉시 이해 가능 | 한 문장은 하나의 메시지만 전달       |
| **Concise (간결한)**    | 핵심 정보만 효율적으로 전달      | 의미 없는 단어/문장 제거          |
| **Casual (친근한)**     | 금융/기술 업계 특유의 딱딱함 탈피  | 해요체 통일, 전문용어 금지         |
| **Respect (존중하는)**   | 단기 전환보다 사용자 신뢰를 우선   | 강요/공포 표현 금지, 정직한 커뮤니케이션 |
| **Emotional (공감하는)** | 정보 전달을 넘어 감정에 공감     | 성취 인정, 상황 공감 메시지        |


### 1-2. 8가지 라이팅 프린시플 `[확인됨 - toss.tech]`

코어밸류를 실제 문장에 녹여내기 위한 8가지 실행 원칙이다.

#### (1) Predictable Hint — 다음 화면을 예상할 수 있는 힌트

사용자가 버튼을 누르기 전에 다음에 무엇이 일어나는지 예측할 수 있어야 한다.


| ❌ Before            | ✅ After                 | 이유                         |
| ------------------- | ----------------------- | -------------------------- |
| "혜택 받기" (막연한 이익 표현) | "쿠폰 3장 받으러 가기" (구체적 내용) | 다음 화면에서 무엇을 볼 수 있는지 명확히 전달 |
| "자세히 보기"            | "이번 달 근무 기록 보기"         | 구체적인 컨텍스트 제공               |


#### (2) Weed Cutting — 의미 없는 단어 모두 제거

모바일에서 사용자는 읽지 않고 스캔한다. 의미에 기여하지 않는 단어를 모두 잘라낸다.


| ❌ Before                       | ✅ After            | 이유                   |
| ------------------------------ | ------------------ | -------------------- |
| "이 서비스를 한번 사용해보시는 것도 좋을 것 같아요" | "이 서비스를 사용해보세요"    | 불필요한 완곡 표현 제거        |
| "앞으로 출근 기록을 확인할 수 있어요"         | "출근 기록을 확인할 수 있어요" | "앞으로" 제거 — 이미 맥락상 명확 |
| "현재 등록된 직원이 없습니다"              | "등록된 직원이 없어요"      | "현재" 제거 + 해요체 전환     |


#### (3) Remove Empty Sentences — 의미 없는 문장 제거

한 화면 안에서 같은 내용을 반복하거나, 장식적인 문장을 넣지 않는다.


| ❌ Before                                           | ✅ After        | 이유                 |
| -------------------------------------------------- | -------------- | ------------------ |
| "환영합니다! 출근 기록을 관리해보세요. 아래에서 오늘의 출근 현황을 확인할 수 있어요." | "오늘의 출근 현황이에요" | 인사+설명+안내의 3중 반복 제거 |


#### (4) Focus on Key Message — 중요한 메시지만 전달

사용자가 지금 알아야 하는 것만 말한다. 이미 아는 맥락까지 설명하지 않는다.


| ❌ Before                                    | ✅ After         | 이유                         |
| ------------------------------------------- | --------------- | -------------------------- |
| "출근 기록은 매일의 근무 시간을 추적하는 기능입니다. 아래에서 확인하세요." | "이번 주 근무 기록이에요" | 기능 설명은 불필요 — 사용자는 이미 알고 있음 |


#### (5) Easy to Speak — 말로 했을 때 자연스러운 표현

소리 내어 읽었을 때 어색하지 않은 문장이어야 한다. 문어체, 전문용어, 부자연스러운 문장 구조를 피한다.


| ❌ Before          | ✅ After                  | 이유                 |
| ----------------- | ------------------------ | ------------------ |
| "생체 인증 수단 미등록 상태" | "생체 인증이 등록되지 않았어요"       | 명사 나열 → 자연스러운 문장   |
| "근태 이상 건 발생"      | "근무 기록에 확인이 필요한 부분이 있어요" | 한자어 명사 나열 → 구어체 전환 |
| "위치정보 취득 불가"      | "위치를 가져올 수 없어요"          | 기술 용어 → 일상어        |


#### (6) Suggest than Force — 강요가 아닌 제안

행동을 강요하거나 공포감을 조성하지 않는다. 선택의 자율성을 존중한다.


| ❌ Before               | ✅ After                  | 이유            |
| ---------------------- | ------------------------ | ------------- |
| "지금 출근하지 않으면 지각 처리돼요!" | "출근 시간이 곧 마감돼요"          | 공포 유발 → 사실 전달 |
| "반드시 알림을 켜주세요"         | "알림을 켜면 출퇴근 시간을 놓치지 않아요" | 강요 → 혜택 제안    |


#### (7) Universal Words — 모두가 이해하는 단어

연령, 학력, 직군에 관계없이 누구나 이해할 수 있는 단어를 사용한다. 유행어, 신조어, 업계 전문용어를 피한다.


| ❌ Before         | ✅ After           | 이유            |
| ---------------- | ----------------- | ------------- |
| "GPS 기반 지오펜싱 인증" | "위치로 출근 확인"       | 기술 용어 제거      |
| "온보딩 프로세스"       | "시작하기" 또는 "처음 설정" | 영어 업무용어 → 일상어 |
| "리텐션 push"       | "다시 방문 알림"        | 마케팅 용어 제거     |


#### (8) Find Hidden Emotion — 숨겨진 감정 찾기

단순 정보 전달을 넘어, 사용자의 감정적 맥락을 인정한다.


| ❌ Before         | ✅ After                    | 이유               |
| ---------------- | -------------------------- | ---------------- |
| "이번 달 근무를 완료했어요" | "이번 달도 수고했어요"              | 성취를 인정하는 공감 메시지  |
| "서류 제출이 완료됐어요"   | "서류를 잘 제출했어요! 검토 후 알려드릴게요" | 노력 인정 + 다음 단계 안내 |

## 2. 어조 (Tone of Voice)

### 2-1. 말투 체계 — `~해요` 체 통일 `[확인됨 - toss.tech, developers-apps-in-toss]`

토스는 전 제품에서 `~해요` 체(해요체)를 사용한다. `~합니다/~했습니다` 습니다체는 사용하지 않는다. 에러 메시지에서도 예외 없이 해요체를 적용한다.


| 상황     | ✅ 올바른 표현     | ❌ 잘못된 표현       | 이유        |
| ------ | ------------ | -------------- | --------- |
| 저장 완료  | "저장했어요"      | "저장되었습니다"      | 해요체 통일    |
| 수정 완료  | "수정했어요"      | "수정되었습니다"      | 해요체 통일    |
| 데이터 없음 | "기록이 없어요"    | "기록이 없습니다"     | 해요체 통일    |
| 권한 없음  | "접근 권한이 없어요" | "접근 권한이 없습니다"  | 해요체 통일    |
| 알림 없음  | "새 알림이 없어요"  | "새로운 알림이 없습니다" | 해요체 + 간결화 |
| 송금 완료  | "보냈어요"       | "이체가 완료되었습니다"  | 구어체 + 해요체 |


### 2-2. 능동형 문장 우선 `[확인됨 - developers-apps-in-toss]`

수동형보다 능동형을 우선 사용한다. `됐어요`보다 `했어요`를 쓴다.


| ✅ 능동형   | ❌ 수동형   | 이유           |
| ------- | ------- | ------------ |
| "저장했어요" | "저장됐어요" | 행동 주체가 명확    |
| "등록했어요" | "등록됐어요" | 사용자의 행동을 인정  |
| "삭제했어요" | "삭제됐어요" | 사용자가 한 일로 표현 |


**수동형이 허용되는 예외 상황:**

- 서비스 종료/만료 안내: "서비스가 종료됐어요" (사용자 행동이 아님)
- 사용자에게 영향이 가는 상황: "비밀번호가 변경됐어요" (타인에 의한 변경)
- 안심 메시지: "안전하게 처리됐어요"

### 2-3. 긍정형 문장 우선 `[확인됨 - developers-apps-in-toss]`

부정적 표현을 최소화하고, 가능하면 긍정형으로 프레이밍한다.


| ❌ 부정형            | ✅ 긍정형                  | 이유          |
| ---------------- | ---------------------- | ----------- |
| "출근할 수 없어요"      | "매장 근처에서 출근할 수 있어요"    | 가능한 행동으로 전환 |
| "수정할 수 없어요"      | "관리자에게 요청하면 수정할 수 있어요" | 해결 방법 제시    |
| "사용할 수 없는 기능이에요" | "이 기능은 곧 사용할 수 있어요"    | 미래 가능성 제시   |


**부정형이 허용되는 예외 상황:**

- 정책상 서비스 이용이 불가능한 경우: "이 기능은 사용할 수 없어요"
- 특정 기능이 명확히 제공되지 않는 경우: "이 기기에서는 지원하지 않아요"
- 사용자의 선택에 대한 결과 경고: "삭제하면 복구할 수 없어요"

### 2-4. 과도한 높임 제거 `[확인됨 - developers-apps-in-toss]`

친근한 어조를 유지하기 위해 과도한 경어 표현을 제거한다.


| ❌ 과도한 높임      | ✅ 적절한 표현  | 이유        |
| ------------- | --------- | --------- |
| "확인하시겠습니까?"   | "확인할까요?"  | 과도한 격식 제거 |
| "입력해주시기 바랍니다" | "입력해주세요"  | 간결한 요청    |
| "로그인하시겠습니까?"  | "로그인할까요?" | 친근한 어조    |
| "계시다"         | "있다"      | 구어체 수준으로  |
| "여쭈다"         | "묻다/물어보다" | 과도한 겸양 제거 |
| "~께"          | "~에게"     | 자연스러운 조사  |


**높임이 허용되는 예외:**

- 사용자 상황을 추측하는 질문: "혹시 ~하셨나요?"
- 설문/피드백 요청: "의견을 들려주시겠어요?"

### 2-5. 주어 최소화 `[패턴 추출 - 다수 소스]`

문맥상 명확한 주어는 생략하여 간결함을 높인다.


| ❌ Before            | ✅ After     | 이유              |
| ------------------- | ----------- | --------------- |
| "정보가 성공적으로 저장되었습니다" | "저장했어요"     | 주어 + "성공적으로" 제거 |
| "직원 정보가 삭제되었습니다"    | "삭제했어요"     | 주어 제거 + 해요체     |
| "데이터를 불러오고 있습니다"    | "불러오는 중이에요" | 주어 생략 + 구어체     |


### 2-6. 사용자 지칭 `[패턴 추출 - 다수 소스]`

- 사용자를 지칭할 때는 **이름+님** 형태 우선: "정표님"
- 부득이한 경우 문맥에 맞게 생략
- "고객님"은 금융 서비스 맥락에서만 사용, HR 앱에서는 "이름+님" 또는 생략

### 2-7. 한자어 명사 나열 금지 `[확인됨 - developers-apps-in-toss]`

한자어 명사를 나열하면 딱딱해진다. `명사+동사` 구조로 바꾸거나 풀어서 쓴다.


| ❌ 명사 나열    | ✅ 풀어쓴 표현             | 이유           |
| ---------- | -------------------- | ------------ |
| "근태 이상 발생" | "근무 기록에 확인할 부분이 있어요" | 자연스러운 문장화    |
| "위치 인증 실패" | "위치를 확인할 수 없었어요"     | 기술 용어 → 구어체  |
| "서류 제출 완료" | "서류를 제출했어요"          | 명사형 → 동사형 문장 |


### 2-8. 표기 규칙 `[패턴 추출 - 다수 소스]`

- `되어요` 대신 `돼요` 사용 (모바일 공간 효율)
- `~었` 과거 시제 마커는 가능하면 제거하여 간결하게
- 느낌표(`!`)는 성공 토스트에 1개까지만 허용, 에러에는 사용 금지

## 3. 컴포넌트별 텍스트 규칙

### 3.1 버튼 (Button)

#### 기본 원칙 `[확인됨 - toss.tech, tossfeed]`

버튼은 **동사형**으로 작성하며, 행동을 명확히 표현한다.

**"하기" vs "할게요" vs "했어요" 구분:**


| 패턴          | 용도                  | 예시                        |
| ----------- | ------------------- | ------------------------- |
| `~하기`       | 사용자가 능동적으로 선택하는 행동  | "저장하기", "출근하기", "등록하기"    |
| `~할게요`      | 사용자의 의지/동의 표현       | "동의할게요", "삭제할게요", "시작할게요" |
| `~했어요`      | 결과 확인 후 닫는 버튼       | "확인했어요", "읽었어요"           |
| `확인` / `다음` | 진행 버튼 (다음 단계로 넘어가는) | 단계별 플로우의 진행 CTA           |
| `취소`        | 닫기/해제 (보조 행동)       | "취소"를 "닫기"보다 우선 사용        |


#### Before/After 예시


| 상황      | ❌ Before     | ✅ After   | 이유            |
| ------- | ------------ | --------- | ------------- |
| 기능 실행   | "출근"         | "출근하기"    | 동사형으로 행동 명확화  |
| 데이터 저장  | "저장", "완료"   | "저장하기"    | 행동의 구체적 표현    |
| 동의/확정   | "동의"         | "동의할게요"   | 사용자 의지 표현     |
| 확인 후 닫기 | "확인", "OK"   | "확인했어요"   | 결과 확인의 뉘앙스    |
| 보조 행동   | "취소", "Skip" | "나중에 할게요" | 친근한 거절 표현     |
| 삭제 확인   | "삭제"         | "삭제할게요"   | 사용자 의지를 담은 확인 |


#### 바텀시트/모달 제목 규칙 `[확인됨 - tossfeed]`

- 바텀시트 제목은 **최대 1줄**
- 행동 유발 버튼: `~하기` 형식
- 진행 버튼: `확인` 또는 `다음`
- 해제 버튼: `취소` ("닫기"보다 우선)

#### 예외

공간이 매우 좁은 경우(탭, 뱃지 등) 명사형 허용: "출근", "퇴근", "알림"

### 3.2 에러 메시지 (Error Message) `[확인됨 - toss.tech]`

#### 6가지 원칙

토스가 공식 발표한 좋은 에러 메시지의 6가지 원칙:


| #   | 원칙                     | 설명                                          |
| --- | ---------------------- | ------------------------------------------- |
| 1   | **최고의 에러는 발생하지 않는 것**  | 에러 메시지 작성 전, UX 개선으로 에러 자체를 방지할 수 있는지 먼저 검토 |
| 2   | **적절한 컴포넌트 선택**        | 설명이 필요한 에러는 다이얼로그, 가벼운 피드백은 토스트, 필드 에러는 인라인 |
| 3   | **스스로 해결할 수 있는 방법 안내** | 에러 상황만 알리지 말고 사용자가 직접 해결할 수 있는 구체적 방법 제시    |
| 4   | **사용자 언어로 쓰기**         | 개발 용어, IT 전문용어 대신 사용자가 이해할 수 있는 일상어 사용      |
| 5   | **쉽게 해결할 수 있게 도와주기**   | "고객센터 문의" 텍스트 대신 탭 한 번으로 연결되는 버튼 제공         |
| 6   | **부정적 감정 최소화**         | "안 돼요", "없어요" 같은 부정형 → "~할 수 있어요" 긍정형 전환    |


#### 에러 메시지 3요소 구조

모든 에러 메시지는 다음 세 가지를 포함해야 한다:

```
① 상황 설명 — 무엇이 문제인지
② 이유 — 왜 발생했는지
③ 해결책 — 어떻게 해결하는지
```

#### Before/After 예시


| ❌ Before     | ✅ After                        | 이유           |
| ------------ | ------------------------------ | ------------ |
| "오류가 발생했습니다" | "저장에 실패했어요 — 다시 시도해주세요"        | 상황+해결책, 해요체  |
| "비밀번호 오류"    | "비밀번호는 8자 이상이어야 해요"            | 구체적 조건 안내    |
| "출근 실패"      | "매장 근처가 아니에요 — 약 150m 거리에 있어요" | 상황+수치적 안내    |
| "위치 오류"      | "위치를 가져올 수 없어요 — Wi-Fi를 켜보세요"  | 이유+해결책       |
| "알 수 없음"     | "신분증 유효기간이 만료됐어요"              | 사용자 언어로 전환   |
| "생체 인증 실패"   | "생체 인증이 안 됐어요 — 앱을 다시 시작해보세요"  | 해결 방법 제시     |
| "네트워크 에러"    | "인터넷 연결을 확인해주세요"               | 기술 용어 제거+해결책 |


#### 에러 컴포넌트 선택 기준


| 에러 유형         | 적절한 컴포넌트             | 부적절한 컴포넌트 |
| ------------- | -------------------- | --------- |
| 입력 필드 유효성     | 인라인 텍스트 (필드 아래)      | 토스트, 모달   |
| 서버 요청 실패      | 토스트 (description 포함) | alert()   |
| 복구 불가능한 중요 에러 | 다이얼로그/바텀시트           | 토스트       |
| 권한 부족         | 전체 화면 또는 바텀시트        | 인라인 텍스트   |


### 3.3 토스트 (Toast)

#### 기본 규칙 `[패턴 추출 - 다수 소스]`

```tsx
// 성공 — 구체적인 결과 명시
toast.success("연경당 카페로 출근했어요")

// 에러 — 이유 + 해결 방법 (description 필수)
toast.error("저장에 실패했어요", {
  description: "잠시 후 다시 시도해주세요"
})

// 경고 — 현재 상태 설명
toast.warning("매장 근처가 아니에요", {
  description: "연경당 카페에서 약 150m 거리에 있어요"
})
```

#### Before/After 예시


| ❌ Before                    | ✅ After                                                       | 이유                        |
| --------------------------- | ------------------------------------------------------------- | ------------------------- |
| `toast.error("저장 실패")`      | `toast.error("저장에 실패했어요", { description: "잠시 후 다시 시도해주세요" })` | 단어만 넣기 금지, description 필수 |
| `toast.success("Success!")` | `toast.success("저장했어요")`                                      | 영어 금지, 해요체                |
| `toast("완료되었습니다")`          | `toast.success("출근했어요!")`                                     | 습니다체 → 해요체, 구체적 행동 명시     |


#### 규칙 정리

- `toast.error`에는 반드시 `description` 포함
- 성공 토스트에 느낌표(`!`)는 **1개까지만** 허용
- 에러 토스트에는 느낌표 사용 금지
- 표시 시간: 짧은 메시지 2초, 긴 메시지 3초
- 스타일: 배경 `#191F28` (다크) + 흰색 텍스트

### 3.4 플레이스홀더 (Placeholder) `[확인됨 - brunch.co.kr/@kellypoly/62]`

#### 기본 원칙

토스는 플레이스홀더에 **질문형 문장**을 사용하여 대화하는 느낌을 준다.


| ❌ Before     | ✅ After          | 이유          |
| ------------ | ---------------- | ----------- |
| "이름 입력"      | "이름을 입력해주세요"     | 문장형으로 안내    |
| "검색"         | "직원 이름으로 검색해보세요" | 구체적 맥락 제공   |
| "0원" (금액 입력) | "얼마나 보낼까요?"      | 대화형 질문으로 전환 |
| "받는 분"       | "어디로 돈을 보낼까요?"   | 사용자 관점 프레이밍 |
| "YYYY-MM-DD" | "날짜를 선택해주세요"     | 포맷 코드 → 자연어 |


#### 규칙 정리

- 기술적 포맷 표기(`YYYY-MM-DD`, `HH:mm`) 대신 자연어 사용
- 예시값을 보여줄 때는 실제 가능한 값 사용: "예: 홍길동"
- 색상: `text-[#B0B8C1]` (text-disabled 토큰)

### 3.5 빈 화면 (Empty State)

#### 기본 원칙 `[패턴 추출 - 다수 소스]`

단순한 "없음" 대신 **상황을 공감하는 문장**을 쓴다. 이모지는 맥락에 맞게 1개 허용.


| ❌ Before   | ✅ After                      | 이유               |
| ---------- | ---------------------------- | ---------------- |
| "데이터 없음"   | "이 날은 출근 기록이 없어요"            | 구체적 상황 공감        |
| "기록이 없습니다" | "기록된 근무 내역이 없어요"             | 해요체 + 친근함        |
| "알림 없음"    | "새 알림이 없어요"                  | 문장형              |
| "결과 없음"    | "검색 결과가 없어요 — 다른 이름으로 찾아보세요" | 상황+다음 행동 안내      |
| "직원 없음"    | "아직 등록된 직원이 없어요"             | "아직"으로 향후 가능성 암시 |


#### 빈 화면 구성 요소

```
[일러스트 또는 아이콘 (선택)]
[공감 메시지 — 해요체 문장]
[다음 행동 안내 — CTA 버튼 또는 설명 (선택)]
```

### 3.6 모달/다이얼로그 (Modal/Dialog)

#### 절대 규칙 `[확인됨 - toss.tech]`

`**alert()`, `confirm()`, `window.prompt()` 사용 절대 금지**

```tsx
// ❌ 절대 금지
if (confirm("삭제하시겠어요?")) { ... }

// ✅ 올바른 방법 — 커스텀 confirm 바텀시트 사용
const [showConfirm, setShowConfirm] = useState(false);
```

#### 컴포넌트 선택 기준


| 상황                       | 사용할 컴포넌트         |
| ------------------------ | ---------------- |
| 확인이 필요한 중요 행동 (삭제, 로그아웃) | 바텀시트 with 버튼 2개  |
| 성공/실패 피드백                | `toast` (sonner) |
| 짧은 선택 (예/아니오)            | 바텀시트 with 버튼 2개  |
| 설명이 필요한 에러               | 다이얼로그 (토스트 아닌)   |
| **목록에서 항목 선택**           | **모달 (화면 중앙)**   |
| **흐름이 막혀 선택이 필수인 경우**    | **모달 (화면 중앙)**   |
| 보조 정보 입력, 사유 작성          | 바텀시트 (아래에서 올라옴)  |


#### 바텀시트 vs 모달 구분 원칙


| 구분     | 바텀시트                    | 모달 (중앙)             |
| ------ | ----------------------- | ------------------- |
| **형태** | 화면 아래에서 올라옴             | 화면 중앙에 띄움           |
| **용도** | 빠른 확인, 짧은 입력, 보조 행동     | 목록 선택, 흐름 차단형 결정    |
| **예시** | 출장출근 확인, 원격퇴근 사유, 삭제 확인 | 매장 선택, 체크리스트, 권한 안내 |
| **닫기** | 백드롭 클릭 또는 취소 버튼         | 백드롭 클릭 + X 버튼 필수    |


#### 모달 텍스트 구조

```
[제목 — 최대 1줄, 핵심 질문/상황]
[본문 — 부가 설명, 결과 안내 (선택)]
[Primary 버튼 — "~하기" 또는 "~할게요"]
[Secondary 버튼 — "취소" 또는 "나중에 할게요"]
```

#### Before/After 예시


| ❌ Before                          | ✅ After                                     | 이유             |
| --------------------------------- | ------------------------------------------- | -------------- |
| 제목: "확인" / 버튼: "예" / "아니오"        | 제목: "이 직원을 삭제할까요?" / 버튼: "삭제할게요" / "취소"     | 구체적 행동 명시      |
| 제목: "경고" / 본문: "이 작업은 되돌릴 수 없습니다" | 제목: "삭제하면 복구할 수 없어요" / 본문: "출근 기록도 함께 삭제돼요" | 해요체 + 영향 범위 안내 |
| 버튼: "OK" / "Cancel"               | 버튼: "확인했어요" / "취소"                          | 영어 금지, 해요체     |


### 3.7 로딩 상태 (Loading State)

#### 기본 원칙 `[패턴 추출 - 다수 소스]`

데이터 로딩 중일 때 빈 화면 또는 텍스트 "로딩 중..." 표시 금지. 항상 실제 레이아웃을 반영한 **Skeleton UI** 제공.

```tsx
// ❌ 금지
if (loading) return <div>로딩 중...</div>

// ✅ 올바른 방법
{loading ? (
  <div className="h-9 w-32 bg-slate-100 animate-pulse rounded-lg" />
) : (
  <p>{data}</p>
)}
```

#### 로딩 텍스트 규칙 (Skeleton이 불가능한 경우)


| ❌ Before     | ✅ After           | 이유            |
| ------------ | ----------------- | ------------- |
| "로딩 중..."    | "데이터를 분석 중이에요..." | 무엇을 하는지 구체적   |
| "Loading..." | "위치를 확인하는 중이에요"   | 영어 금지, 구체적 행동 |
| "잠시만요"       | "불러오는 중이에요..."    | 현재 상태 명시      |
| "처리 중..."    | "출근을 기록하는 중이에요"   | 사용자 행동과 연결    |


#### 버튼 로딩 상태

```tsx
// ✅ 버튼 내 스피너 + 텍스트 변경 + disabled
<button disabled className="...">
  <Spinner className="w-4 h-4 animate-spin" />
  저장하는 중...
</button>
```

## 4. 피해야 할 표현 (Anti-Patterns) `[확인됨 - toss.tech, developers-apps-in-toss]`

### 4-1. 절대 금지 표현


| 카테고리    | ❌ 금지 표현                | ✅ 대체 표현              | 이유                 |
| ------- | ---------------------- | -------------------- | ------------------ |
| 말투      | `~합니다`, `~했습니다`        | `~해요`, `~했어요`        | 해요체 통일 원칙          |
| 시스템 언어  | "Error", "404", "null" | 자연어로 설명              | 사용자 언어 원칙          |
| 브라우저 기본 | `alert()`, `confirm()` | 커스텀 모달/토스트           | 일관된 UX             |
| 과도한 높임  | "~하시겠습니까?"             | "~할까요?"              | 친근한 어조             |
| 공포 유발   | "지금 안 하면 ~"            | "~하면 ~할 수 있어요"       | Suggest than Force |
| 영어 혼용   | "OK", "Cancel", "Skip" | "확인했어요", "취소", "나중에" | 한글 통일              |


### 4-2. 문체 Anti-Patterns


| ❌ Anti-Pattern | ✅ Pattern | 이유                              |
| -------------- | --------- | ------------------------------- |
| 관료적/공문서체       | 구어체       | "귀하의 요청이 접수되었습니다" → "요청했어요"     |
| 이중 부정          | 단순 긍정     | "불가능하지 않습니다" → "가능해요"           |
| 기술 용어 노출       | 일상어       | "timeout 발생" → "시간이 오래 걸리고 있어요" |
| 수동형 남용         | 능동형       | "처리되었습니다" → "처리했어요"             |
| 명사 나열          | 동사형 문장    | "등록 완료" → "등록했어요"               |
| 감정 없는 정보 전달    | 공감 메시지    | "대출 잔액 감소" → "갚느라 고생했어요"        |


### 4-3. 콘텐츠 Anti-Patterns `[확인됨 - Simplicity 23]`


| ❌ Anti-Pattern      | ✅ Pattern     | 이유                  |
| ------------------- | ------------- | ------------------- |
| 이모지/특수문자로 궁금증 유발    | 핵심 내용을 직접 전달  | 기대와 실제의 괴리 방지       |
| "이것은 무엇일까요?" 식 수수께끼 | 구체적 수치와 사실 제시 | Predictable Hint 원칙 |
| 공급자 관점 작성           | 사용자 관점 작성     | 사용자가 얻는 가치 중심       |

## 5. 포맷팅 규칙

### 5-1. 숫자와 단위


| 유형  | ✅ 올바른 표기          | ❌ 잘못된 표기                 |
| --- | ----------------- | ------------------------ |
| 금액  | "50,000원"         | "50000원", "5만원"          |
| 거리  | "150m"            | "150미터", "0.15km"        |
| 시간  | "9:00" 또는 "오전 9시" | "09:00:00", "9시 00분"     |
| 날짜  | "3월 18일 (수)"      | "2026-03-18", "26/03/18" |
| 퍼센트 | "95%"             | "95퍼센트"                  |
| 인원  | "3명"              | "3 명" (숫자와 단위 사이 공백 없음)  |


### 5-2. 문장 부호


| 규칙                    | 예시                         |
| --------------------- | -------------------------- |
| 마침표: UI 텍스트에서는 **생략** | "저장했어요" (O) / "저장했어요." (X) |
| 물음표: 질문형에만 사용         | "삭제할까요?"                   |
| 느낌표: 성공 메시지에 1개까지     | "출근했어요!"                   |
| 말줄임표: 로딩/진행 중에만       | "불러오는 중이에요..."             |
| 대시(—): 부가 설명 연결       | "저장에 실패했어요 — 다시 시도해주세요"    |


### 5-3. 띄어쓰기

- 숫자+단위: 붙여쓰기 ("150m", "3명", "50,000원")
- 대시 전후: 공백 포함 (" — ")
- 괄호 전후: 한글 뒤 공백 후 괄호 ("3월 18일 (수)")

| `bg-white`       | `#FFFFFF` | 카드, 입력 배경                  |
| `border`         | `#E5E8EB` | 카드 테두리, 구분선                |
| `primary-light`  | `#E8F3FF` | 파란색 배경 강조 (알림 뱃지, 출근 중 상태) |


#### 상태 컬러


| 상태      | 배경        | 텍스트       |
| ------- | --------- | --------- |
| 성공/진행 중 | `#E8F3FF` | `#3182F6` |
| 경고/이상   | `#FFF4E6` | `#D9480F` |
| 에러      | `#FFEBEB` | `#E03131` |
| 비활성     | `#F2F4F6` | `#8B95A1` |


#### 색상 사용 금지 규칙

- 배경에 포화도 높은 색 사용 금지 (배경은 항상 연한 파스텔 or 흰색)
- 테두리 색으로 primary(`#3182F6`) 직접 사용 금지 — opacity 적용 (`#3182F6/30`)
- 텍스트와 배경 간 대비율 WCAG AA 기준 준수 (4.5:1 이상)

### 6-2. 타이포그래피

**Pretendard** 단일 사용.

```css
font-family: 'Pretendard', -apple-system, BlinkMacSystemFont, sans-serif;
```

### 6-3. 컴포넌트 스타일

#### 버튼

```
Primary:     h-14(56px), rounded-2xl, bg-[#3182F6], text-white, font-bold text-lg
Secondary:   h-14(56px), rounded-2xl, bg-[#E8F3FF], text-[#3182F6], border
Destructive: h-14(56px), rounded-2xl, bg-[#FFEBEB], text-[#E03131]
Ghost:       h-10(40px), rounded-xl, bg-transparent or bg-[#F2F4F6]
```

- 비활성: `disabled:bg-[#D1D6DB] disabled:text-white disabled:cursor-not-allowed`
- 로딩 중: 버튼 내 스피너 + 텍스트 변경 + disabled
- 탭 피드백: `active:scale-[0.97]`
- 아이콘+텍스트: 아이콘은 텍스트 크기의 80%

#### 입력 필드

```
h-14(56px), rounded-xl, border border-[#E5E8EB]
focus: border-[#3182F6] ring-0
error: border-[#E03131]
placeholder: text-[#B0B8C1]
```

- 레이블: 인풋 위, `text-[13px] font-medium text-[#4E5968]`
- 에러: 인풋 아래, `text-[13px] text-[#E03131]` 인라인
- 성공/검증: border 초록 계열 변경

#### 카드

```
bg-white, rounded-[28px], border border-slate-100, shadow-sm, p-5 or p-6
hover: shadow-[0_4px_20px_rgba(0,0,0,0.06)] transition-all
```

- 내부 구분: `border-t border-[#F2F4F6]`
- 클릭 시: `hover:bg-[#F9FAFB]`

### 6-5. 인터랙션 & 애니메이션

```css
transition-all duration-200     /* 기본 상태 전환 */
transition-all duration-300     /* 색상 변화 */
```

- 버튼 탭 시: `active:scale-[0.97]`

## 7. 실전 적용 체크리스트

기능 구현 전 이 체크리스트를 확인한다.

### UX 라이팅

- 모든 문구가 `~해요` 체인가?
- 습니다체(`~합니다`, `~했습니다`)가 사용되지 않았는가?
- 능동형 문장을 우선 사용했는가? (`했어요` > `됐어요`)
- 부정형 대신 긍정형으로 프레이밍했는가?
- 과도한 높임 표현(`~하시겠습니까?`)이 없는가?
- 한자어 명사 나열 없이 자연스러운 문장인가?
- 전문용어/기술용어/영어가 사용자에게 노출되지 않는가?
- 소리 내어 읽었을 때 자연스러운가? (Easy to Speak)

### 컴포넌트 텍스트

- 버튼 텍스트가 동사형(`~하기`, `~할게요`, `~했어요`)인가?
- 에러 토스트에 `description`이 포함돼 있는가?
- 에러 메시지에 해결 방법이 안내돼 있는가?
- 빈 화면(Empty State)에 공감 메시지가 있는가?
- 플레이스홀더가 구체적이고 자연어인가?
- 모달 제목이 1줄 이내이고 구체적인가?
- 로딩 상태에 Skeleton UI가 있는가?

### 디자인 시스템

- `alert()`, `confirm()`, `window.prompt()`를 사용하지 않았는가?
- 색상이 팔레트 토큰 내에 있는가?
- 폰트는 Pretendard만 사용하는가?
- 반응형(모바일 360px+) 대응이 됐는가?
- 중요 파괴적 행동(삭제 등)에 커스텀 확인 단계가 있는가?
- 터치 타겟이 44x44px 이상인가?
- 아이콘 전용 버튼에 `aria-label`이 있는가?

### 감정/공감

- 성취 상황에서 사용자의 노력을 인정했는가? (Find Hidden Emotion)
- 에러 상황에서 사용자 탓이 아님을 암시하는 문체인가?
- 강요나 공포감 없이 제안하는 어조인가? (Suggest than Force)
