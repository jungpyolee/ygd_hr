# 토스(Toss) UI/UX 가이드라인

> **목적**: ygd_hr 프로젝트의 모든 UI/UX 개발 기준 문서
> **기반**: 토스 앱 디자인 원칙 및 UX 라이팅 규칙
> **최종 갱신**: 2026-03-16

---

## 1. 핵심 디자인 원칙

### 1-1. 단순함 (Simplicity)
불필요한 요소는 제거한다. 사용자의 목적 달성에 기여하지 않는 장식, 문구, 버튼은 존재하지 않는다.

- 화면당 핵심 행동(CTA)은 **하나**
- 정보는 계층을 만들어 제공 — 핵심 → 보조 → 상세
- 아이콘만으로 의미가 불명확한 경우 반드시 텍스트 레이블 병기

### 1-2. 명확함 (Clarity)
사용자가 지금 무엇을 할 수 있는지, 무슨 상태인지 항상 알 수 있어야 한다.

- 버튼 비활성화 이유를 UI로 명시 (e.g. 출근 중일 때 출근 버튼 비활성)
- 로딩/에러/빈 상태 모두 별도 UI 처리
- 상태 변화는 즉각적인 피드백 제공

### 1-3. 신뢰 (Trust)
디자인이 서비스에 대한 신뢰를 만든다.

- 일관된 색상/타이포/컴포넌트 패턴 유지
- 에러는 사용자 탓이 아님을 암시하는 문체
- 중요한 행동(삭제, 로그아웃)은 확인 단계를 거침 — **단, 브라우저 기본 다이얼로그 금지**

## 2. UX 라이팅 규칙

### 2-1. 말투 체계 — `~해요` 체 **통일**

토스는 전 제품에서 `~해요` 체를 사용한다. `~합니다/~했습니다` 습니다체는 사용하지 않는다.

| 상황 | 올바른 표현 | 잘못된 표현 |
|------|------------|------------|
| 저장 완료 | "저장했어요" | "저장되었습니다" |
| 수정 완료 | "수정했어요" | "수정되었습니다" |
| 데이터 없음 | "기록이 없어요" | "기록이 없습니다" |
| 권한 없음 | "접근 권한이 없어요" | "접근 권한이 없습니다" |
| 알림 없음 | "새 알림이 없어요" | "새로운 알림이 없습니다" |

### 2-2. 주어 최소화

굳이 쓰지 않아도 이해되는 주어는 생략한다.

| 올바른 표현 | 잘못된 표현 |
|------------|------------|
| "저장됐어요" | "정보가 성공적으로 저장되었습니다" |
| "삭제됐어요" | "직원 정보가 삭제되었습니다" |
| "불러오는 중이에요" | "데이터를 불러오고 있습니다" |

### 2-3. 구체적이고 친절한 에러 메시지

에러 메시지는 ① 무엇이 문제인지 ② 어떻게 해결하는지를 함께 알려준다.

| 올바른 표현 | 잘못된 표현 |
|------------|------------|
| "비밀번호는 8자 이상이어야 해요" | "비밀번호 오류" |
| "매장 근처가 아니에요 — 약 150m 거리에 있어요" | "출근 실패" |
| "위치 정보를 가져올 수 없어요 — Wi-Fi를 켜보세요" | "위치 오류" |
| "저장에 실패했어요 — 다시 시도해주세요" | "저장 실패" |

### 2-4. CTA 버튼 텍스트

버튼은 **동사형**으로 작성한다. 행동을 명확히 표현한다.

| 상황 | 올바른 표현 | 잘못된 표현 |
|------|------------|------------|
| 기능 실행 | "출근하기" | "출근" |
| 데이터 저장 | "저장하기" | "저장", "완료" |
| 동의/확정 | "동의할게요" | "동의" |
| 확인 후 닫기 | "확인했어요" | "확인", "OK" |
| 보조 행동 | "나중에 할게요" | "취소", "Skip" |

**"하기" vs "할게요" 구분:**
- `~하기` → 사용자가 능동적으로 선택하는 행동 (`"저장하기"`, `"출근하기"`)
- `~할게요` → 사용자의 의지/동의 표현 (`"동의할게요"`, `"삭제할게요"`)
- `~했어요` → 결과 확인 후 닫는 버튼 (`"확인했어요"`)

예외: 공간이 매우 좁은 경우(탭, 뱃지 등) 명사형 허용

### 2-5. 빈 상태(Empty State) 메시지

단순한 "없음" 대신 **상황을 공감하는 문장**을 쓴다. 이모지는 맥락에 맞게 1개 허용.

| 올바른 표현 | 잘못된 표현 |
|------------|------------|
| "이 날은 출근 기록이 없어요" | "데이터 없음" |
| "기록된 근무 내역이 없어요 ☕️" | "기록이 없습니다" |
| "새 알림이 없어요" | "알림 없음" |

### 2-6. 로딩 메시지

로딩 텍스트는 **무엇을 하고 있는지** 구체적으로 표현한다. 단순 "로딩 중..." 금지.

| 올바른 표현 | 잘못된 표현 |
|------------|------------|
| "데이터를 분석 중이에요..." | "로딩 중..." |
| "위치를 확인하는 중이에요" | "Loading..." |
| "불러오는 중이에요..." | "잠시만요" |

### 2-7. 사용자 지칭

- 사용자를 지칭할 때는 **이름+님** 형태 우선
- 부득이한 경우 "고객님" 대신 문맥에 맞게 생략 또는 "님"

### 3-1. 기본 팔레트

| 토큰 | 값 | 용도 |
|------|-----|------|
| `primary` | `#3182F6` | 주요 액션, 링크, 포커스 상태 |
| `text-primary` | `#191F28` | 제목, 핵심 정보 |
| `text-secondary` | `#4E5968` | 보조 텍스트, 아이콘 |

| `bg-default` | `#F2F4F6` | 앱 배경, 비활성 영역 |
| `bg-white` | `#FFFFFF` | 카드, 입력 배경 |
| `border` | `#E5E8EB` | 카드 테두리, 구분선 |
| `primary-light` | `#E8F3FF` | 파란색 배경 강조 (알림 뱃지, 출근 중 상태) |

### 3-2. 상태 컬러

| 상태 | 배경 | 텍스트 |
|------|------|--------|
| 성공/진행 중 | `#E8F3FF` | `#3182F6` |
| 경고/이상 | `#FFF4E6` | `#D9480F` |
| 에러 | `#FFEBEB` | `#E03131` |
| 비활성 | `#F2F4F6` | `#8B95A1` |

### 3-3. 색상 사용 금지 규칙

- 배경에 포화도 높은 색 사용 금지 (배경은 항상 연한 파스텔 or 흰색)
- 테두리 색으로 primary(`#3182F6`) 직접 사용 금지 — opacity 적용 (`#3182F6/30`)
- 텍스트와 배경 간 대비율 WCAG AA 기준 준수 (4.5:1 이상)

## 4. 타이포그래피

### 4-1. 폰트

**Pretendard** 단일 사용. Geist 등 영문 폰트를 레이아웃 폰트로 적용하지 않는다.

```css
font-family: 'Pretendard', -apple-system, BlinkMacSystemFont, sans-serif;
```

### 4-2. 텍스트 스케일

| 용도 | 크기 | 굵기 | Tailwind 예시 |
|------|------|------|--------------|
| 페이지 제목 | 24px | bold (700) | `text-2xl font-bold` |
| 섹션 제목 | 20px | bold (700) | `text-xl font-bold` |
| 카드 제목 | 18px | bold (700) | `text-[18px] font-bold` |
| 본문 강조 | 16px | bold (700) | `text-[16px] font-bold` |
| 본문 | 15px | medium (500) | `text-[15px] font-medium` |
| 보조 텍스트 | 14px | regular (400) | `text-sm` |
| 캡션/레이블 | 13px | medium (500) | `text-[13px] font-medium` |
| 마이크로 텍스트 | 12px | medium (500) | `text-[12px] font-medium` |

### 4-3. 금지 사항

- `font-size: 10px` 이하 사용 금지 (접근성)
- 가변 폰트 없이 `font-weight: 100~300` 사용 금지 (Pretendard 지원 굵기: 400, 500, 600, 700, 800)

## 5. 컴포넌트 규칙

### 5-1. 버튼

```
Primary: h-14(56px), rounded-2xl, bg-[#3182F6], text-white, font-bold text-lg
Secondary: h-14(56px), rounded-2xl, bg-[#E8F3FF], text-[#3182F6], border
Destructive: h-14(56px), rounded-2xl, bg-[#FFEBEB], text-[#E03131]
Ghost: h-10(40px), rounded-xl, bg-transparent or bg-[#F2F4F6]
```

- 비활성 상태: `disabled:bg-[#D1D6DB] disabled:text-white disabled:cursor-not-allowed`
- 로딩 중: 버튼 내 스피너 + 텍스트 변경 (버튼 자체 비활성)
- 아이콘+텍스트 조합 시: 아이콘은 텍스트 크기의 80% (e.g. text-lg + w-5 h-5)

### 5-2. 입력 필드

```
h-14(56px), rounded-xl, border border-[#E5E8EB]
focus: border-[#3182F6] ring-0 (outline 대신 border 색 변경)
error: border-[#E03131]
placeholder: text-[#B0B8C1]
```

- 레이블은 인풋 위에 `text-[13px] font-medium text-[#4E5968]` 으로 표시
- 에러 메시지는 인풋 아래 `text-[13px] text-[#E03131]` 으로 인라인 표시
- 성공/검증 완료 상태는 border를 초록계열로 변경

### 5-3. 카드

```
bg-white, rounded-[28px], border border-slate-100, shadow-sm, p-5 or p-6
hover (선택 가능 카드): shadow-[0_4px_20px_rgba(0,0,0,0.06)] transition-all
```

- 카드 내부 섹션 구분: `border-t border-[#F2F4F6]` (진한 구분선 금지)
- 카드 클릭 시: `hover:bg-[#F9FAFB]` 배경 변화

### 5-4. 모달 / 바텀 시트

**규칙: `alert()`, `confirm()`, `window.prompt()` 사용 절대 금지**

대신 다음을 사용:
- **확인이 필요한 중요 행동** (삭제, 로그아웃 등): 바텀 시트 또는 커스텀 모달
- **성공/실패 피드백**: `toast` (sonner)
- **짧은 선택** (예/아니오): 바텀 시트 with 버튼 2개

```tsx
// ❌ 절대 금지
if (confirm("삭제하시겠어요?")) { ... }

// ✅ 올바른 방법 — 커스텀 confirm 바텀시트 사용
const [showConfirm, setShowConfirm] = useState(false);
// ... 바텀시트 or 모달로 확인 받기
```

### 5-5. 토스트 (sonner)

```tsx
// 성공 — 구체적인 결과 명시
toast.success("연경당 카페로 출근했어요")

// 에러 — 이유 + 해결 방법
toast.error("저장에 실패했어요", {
  description: "잠시 후 다시 시도해주세요"
})

// 경고 — 현재 상태 설명
toast.warning("매장 근처가 아니에요", {
  description: `연경당 카페에서 약 150m 거리에 있어요`
})
```

- `toast.error("저장 실패")` 처럼 단어만 넣는 것 금지
- 성공 토스트에 `!`는 1개까지만 허용

### 5-6. 스켈레톤 로딩

데이터 로딩 중일 때 빈 화면 또는 텍스트 "로딩 중..." 표시 금지.
항상 실제 레이아웃을 반영한 Skeleton UI 제공.

```tsx
// ❌ 금지
if (loading) return <div>로딩 중...</div>

// ✅ 올바른 방법
{loading ? (
  <div className="h-9 w-32 bg-slate-100 animate-pulse rounded-lg" />
) : (
  <p>{data}</p>
)}
```

## 6. 레이아웃 & 스페이싱

### 6-1. 페이지 구조

```
├── Sticky Navigation (z-10, backdrop-blur-md)
├── Main Content (px-5, space-y-4)
│   ├── Header Section (py-6)
│   ├── Primary Card
│   ├── Secondary Cards
│   └── Helper/Info Section
└── (필요시) Bottom Sheet
```

### 6-2. 스페이싱 원칙

| 용도 | 값 |
|------|-----|
| 페이지 수평 패딩 | `px-5` (20px) |
| 카드 내부 패딩 | `p-5` (20px) 또는 `p-6` (24px) |
| 카드 간 간격 | `space-y-4` (16px) |
| 카드 내 섹션 간격 | `gap-3` (12px) 또는 `mb-4` (16px) |
| 인라인 요소 간격 | `gap-2` (8px) |

### 6-3. 반응형

- 모바일 퍼스트 (360px+)
- 테블릿 breakpoint: `md:` (768px)
- 그리드: `grid-cols-1` → `sm:grid-cols-2` 또는 `md:grid-cols-3`

## 7. 인터랙션 & 애니메이션

### 7-1. 전환 효과

```css
transition-all duration-200     /* 기본 상태 전환 */
transition-all duration-300     /* 색상 변화 */
animate-pulse                   /* 스켈레톤 로딩 */
animate-spin                    /* 스피너 */
animate-in fade-in duration-500 /* 페이지 진입 */
```

- 전환 시간 200~300ms 권장. 500ms 이상은 페이지 진입에만 허용.
- `transition: all` 보다 `transition-colors`, `transition-shadow` 등 명시적 사용 권장
- 버튼 탭 시: `active:scale-[0.97]` — 즉각적 눌림 피드백

**토스트 스타일:**
토스트 배경은 `#191F28` (다크) + 흰색 텍스트. sonner 기본 스타일이 아니면 커스터마이즈 필요.
표시 시간: 짧은 메시지 2초, 긴 메시지 3초.

### 7-2. 피드백 패턴

| 상황 | 피드백 방법 |
|------|------------|
| 버튼 클릭 중 | 버튼 텍스트 변경 + disabled + (필요시 스피너) |
| 데이터 저장 성공 | toast.success (3초 자동 닫힘) |
| 데이터 저장 실패 | toast.error + description |
| 폼 유효성 실패 | 인라인 에러 텍스트 (toast 금지) |
| 페이지 로딩 | Skeleton UI |

### 7-3. 제스처

- 바텀 시트: 스와이프 다운으로 닫기 지원
- 목록 아이템: 탭 시 즉각 반응 (100ms 이내)

## 8. 내비게이션

### 8-1. 직원(Employee) 앱 구조

```
/ (메인 홈)
  └── /attendances (출퇴근 기록)
```

- 단순 구조, 하단 탭바 불필요 (페이지 2개 이하)
- 관리자 진입: 홈 상단 "어드민" 버튼

### 8-2. 관리자(Admin) 앱 구조

```
/admin (대시보드)
  ├── /admin/employees (직원 관리)
  └── /admin/attendance (근태 캘린더)
```

- 데스크탑: 왼쪽 사이드바
- 모바일: 하단 바텀 시트 네비게이션
- 현재 페이지 active 상태 명확히 표시

### 8-3. 뒤로가기

- 모달 닫기: 모달 바깥 클릭 또는 X 버튼
- 바텀 시트 닫기: 스와이프 다운 또는 X 버튼
- 페이지 이동: Next.js `router.push()` (history 유지)

## 9. 접근성

- 버튼에는 항상 의미있는 텍스트 또는 `aria-label`
- 아이콘 전용 버튼: `aria-label` 필수
- 색상만으로 상태 표현 금지 (색맹 대응: 텍스트/아이콘 병용)
- 포커스 링: `focus-visible:ring-2 focus-visible:ring-[#3182F6]`
- **최소 터치 타겟: 44×44px** — 시각적으로 작아 보여도 실제 터치 영역은 44px 이상 확보
  ```tsx
  // 아이콘 버튼에 터치 영역 확보
  <button className="w-10 h-10 flex items-center justify-center" aria-label="닫기">
    <X className="w-5 h-5" />
  </button>
  ```
