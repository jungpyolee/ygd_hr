## 1. 사업자 등록

### 1-1. 개인사업자 vs 법인 설립 비교

| 항목 | 개인사업자 | 법인 (주식회사/유한회사) |
|------|-----------|----------------------|
| **설립 비용** | 0원 (세무서 방문/홈택스) | 등록면허세 + 법무사 비용 약 50~100만원 |
| **설립 소요 시간** | 당일 | 1~2주 |
| **세금** | 종합소득세 6~45% (누진) | 법인세 9~24% + 배당소득세 |
| **대표 책임** | 무한 책임 | 유한 책임 (출자액 한도) |
| **신뢰도** | 소규모 인식 | B2B 거래 시 신뢰도 높음 |
| **4대 보험** | 대표는 지역가입자 | 대표도 직장가입자 |
| **투자 유치** | 불가 | 가능 (주식 발행) |
| **운영 부담** | 낮음 (간이과세 가능) | 높음 (복식부기, 결산, 주주총회) |

**권장 전략:**

```
Phase 1 (베타 ~ 유료화 초기): 개인사업자
  - 비용 최소화, 빠른 시작
  - 간이과세자 가능 (연 매출 1억 400만원 미만)
  - 유료 전환 시점에 일반과세자 전환

Phase 2 (월 매출 500만원 이상 또는 투자 유치 시): 법인 전환
  - 세율 이점 발생 시점
  - 배상 리스크 분리 (유한 책임)
  - 투자 유치 구조 마련
```

### 1-2. 통신판매업 신고

**필요 여부: 유료화 시 필수**

- 근거: 「전자상거래 등에서의 소비자보호에 관한 법률」 제12조
- SaaS 구독 판매 = 통신판매에 해당
- 관할 구청/시청에 신고 (온라인 가능: 정부24)
- **무료 베타 기간에는 불필요**, 유료 전환 직전에 신고

**필요 서류:**
- 사업자등록증
- 구매안전서비스 이용 확인증 (에스크로 또는 소비자 피해보상보험)
- 호스팅 서버 소재 확인서 (Vercel/Supabase → 해외 서버이므로 "해외" 기재)

**등록면허세:** 약 40,500원 (지역에 따라 상이)

### 1-3. 부가통신사업자 신고

**필요 여부: 필수**

- 근거: 「전기통신사업법」 제22조
- 타인의 통신설비(AWS/Supabase)를 이용하여 부가통신서비스 제공 = 해당
- 과학기술정보통신부에 신고 (온라인: 정보통신 종합 민원실)
- **무료 서비스도 해당** — 출시 전 신고 권장

**신고 절차:**
1. 정보통신 종합 민원실 접속 (https://eshop.kca.kr)
2. 부가통신사업 신고서 작성
3. 사업자등록증 사본 첨부
4. 처리 기간: 약 7~14일

**주의:** 미신고 시 과태료 최대 3,000만원 (실무적으로 초기 스타트업에 즉시 부과 사례는 드물지만, 법적 의무)

### 1-4. 시점 정리

```
[즉시 — 베타 출시 전]
  1. 개인사업자 등록 (홈택스, 당일)
  2. 부가통신사업자 신고 (7~14일)

[유료화 직전]
  3. 통신판매업 신고 (관할 구청)
  4. 에스크로/소비자피해보상보험 가입
```

### 2-3. 위치정보 수집 동의 (위치정보법)

> 별도 섹션 3에서 상세 기술. 여기서는 개인정보보호법과의 관계만 정리.

- GPS 출퇴근 = 위치정보 수집으로, **개인정보보호법과 위치정보법 양쪽 동의** 필요
- 개인정보처리방침에 위치정보 수집 항목 반드시 명시
- 위치정보 수집 시 별도 동의 화면 필요 (아래 섹션 3 참조)

### 2-4. 제3자 제공 동의

**제3자 제공이 발생하는 경우:**

| 제공받는 자 | 제공 항목 | 제공 목적 | 보유 기간 |
|------------|----------|----------|----------|
| 카카오 (OAuth) | 없음 (카카오 → 출첵 방향만) | - | - |
| Apple (OAuth) | 없음 (Apple → 출첵 방향만) | - | - |
| PG사 (유료화 시) | 결제 정보 | 결제 처리 | PG사 정책에 따름 |
| 사업주 (owner) | 소속 직원의 출퇴근/스케줄/크레딧 | 근태 관리 | 고용 관계 종료 시까지 |

**중요 구분:**
- 카카오/Apple OAuth는 **출첵이 정보를 받는 쪽**이므로 제3자 "제공"이 아님
- 다만 카카오 SDK 사용 시 카카오가 기기 정보를 수집할 수 있으므로 이를 개인정보처리방침에 명시
- 사업주에게 직원 정보 표시 = 서비스 본질적 기능이나, 해당 내용을 이용약관/개인정보처리방침에 명시 필요

**동의 화면 구성:**

```
[필수] 개인정보 수집 및 이용 동의
  └── 수집 항목, 목적, 보유기간 요약 표시
      └── [전문 보기] 링크

[필수] 위치정보 수집 및 이용 동의 (직원용)
  └── GPS 출퇴근 기록 관련 동의
      └── [전문 보기] 링크

[선택] 마케팅 정보 수신 동의
  └── 이메일/푸시 알림
      └── [전문 보기] 링크

[필수] 이용약관 동의
  └── [전문 보기] 링크
```

### 2-5. 개인정보 파기 절차

**파기 사유:**
- 회원 탈퇴 시
- 보유 기간 만료 시
- 서비스 종료 시
- 정보주체의 파기 요청 시

**파기 절차:**

```
1. 파기 대상 확인
   └── 보유 기간 경과 여부 확인 (자동화 배치)

2. 즉시 파기 대상 (탈퇴 시)
   ├── auth.users: 소프트 삭제 (Supabase 정책)
   ├── profiles: 개인 식별 정보 마스킹 (이름 → "탈퇴회원", 이메일 → hash)
   ├── 프로필 사진: Storage에서 물리 삭제
   └── 은행 계좌번호: 즉시 삭제

3. 유예 파기 대상 (탈퇴 후 30일)
   ├── 이메일, 이름, 전화번호
   └── 유예 기간 내 재가입 요청 시 복원 가능

4. 법령 보존 대상 (별도 분리 보관)
   ├── 출퇴근 기록: 3년 (근로기준법 제42조)
   ├── 급여 지급 기록: 3년 (근로기준법 제48조)
   ├── 계약/거래 기록: 5년 (상법 제33조)
   └── 전자상거래 기록: 5년 (전자상거래법 제6조)

5. 파기 방법
   ├── DB 레코드: DELETE 또는 데이터 마스킹 후 물리 삭제
   ├── 파일 (Storage): Supabase Storage에서 물리 삭제
   └── 백업: 백업 보유 기간(30일) 경과 후 자동 삭제
```

**구현 필요 사항:**
- 탈퇴 시 즉시 파기 API 구현
- 보유 기간 만료 데이터 자동 삭제 배치 (월 1회)
- 파기 로그 기록 (audit_logs 활용)

### 2-6. 개인정보보호 책임자 지정

**법적 요건:**
- 「개인정보 보호법」 제31조: 개인정보처리자는 개인정보보호 책임자를 지정해야 함
- 소규모 개인사업자도 예외 없음

**출첵 지정 현황:**

| 항목 | 내용 |
|------|------|
| 개인정보보호 책임자 | 이정표 (대표) |
| 직위 | 대표 / 서비스 운영자 |
| 연락처 | (이메일, 전화번호 기재 — 개인정보처리방침에 공개) |
| 담당 업무 | 개인정보 처리 관련 민원 처리, 피해 구제, 방침 수립 |

**향후 법인 전환 시:**
- 임원급 또는 개인정보 처리에 관한 업무를 총괄하는 부서장을 책임자로 지정
- 연 1회 개인정보보호 교육 이수 의무

## 3. 위치정보법

> 근거: 「위치정보의 보호 및 이용 등에 관한 법률」

### 3-1. 위치기반서비스사업자 신고 필요 여부

**결론: 신고 필요**

출첵의 GPS 출퇴근 기능은 다음에 해당:
- 위치정보를 이용하여 제공하는 서비스 = **위치기반서비스**
- 타인(직원)의 위치정보를 수집하여 사업주에게 제공 = **위치기반서비스사업**

**신고 대상 판단:**

```
Q: 위치정보사업자인가? (측위 인프라 직접 운영)
A: 아니요 → GPS는 기기 자체 기능, 출첵이 측위하지 않음

Q: 위치기반서비스사업자인가? (위치정보를 이용한 서비스 제공)
A: 예 → GPS 좌표 수집하여 출퇴근 인증 서비스 제공

∴ "위치기반서비스사업자" 신고 필요
```

**신고 절차:**
1. 방송통신위원회에 신고 (온라인 가능)
2. 신고서 + 위치정보 이용/제공 사실 확인자료 + 위치정보 보호조치 계획 제출
3. 사업자등록증 사본 첨부
4. 처리 기간: 약 30일
5. 비용: 무료

**미신고 시 제재:** 과태료 최대 3,000만원

### 3-2. GPS 출퇴근 기록 = 위치정보 수집

**위치정보법상 정의:**

```
"개인위치정보" = 특정 개인의 위치정보
"위치정보"     = 이동성 있는 물건 또는 개인이 특정 시간에 존재하거나 존재하였던 장소에 관한 정보

→ 출첵의 GPS 출퇴근 기록 = 개인위치정보에 명확히 해당
```

**수집되는 위치 데이터:**
- 출근 시 GPS 좌표 (위도, 경도)
- 퇴근 시 GPS 좌표
- 출퇴근 판정을 위한 매장 위치와의 거리 계산
- 타임스탬프 (위치 + 시간 = 이동 경로 추론 가능)

**법적 의무:**
- 개인위치정보 수집 전 반드시 동의 획득 (법 제18조)
- 수집 목적 외 이용 금지 (법 제19조)
- 위치정보 보유 기간 명시 및 기간 경과 후 파기 (법 제23조)
- 위치정보 이용/제공 사실 확인 자료 6개월 보관 (법 제16조)

## 4. 근로기준법 관련

> 출첵은 근태 관리 "도구"이지 법적 사용자(사업주)가 아님. 그러나 도구 제공자로서 법 준수를 지원하는 기능을 올바르게 구현해야 함.

### 4-1. 출퇴근 기록 보관 의무 (3년)

**법적 근거:**
- 「근로기준법」 제42조: 사용자는 근로자 명부와 대통령령으로 정하는 근로계약에 관한 중요 서류를 3년간 보존
- 시행령 제22조: 출근부 포함

**출첵이 해야 할 것:**

| 의무 | 구현 방안 |
|------|----------|
| 출퇴근 기록 3년 보관 | `attendance_logs` 테이블 데이터 최소 3년 보존 정책 |
| 삭제 방지 | owner도 출퇴근 기록 삭제 불가 (수정만 가능, 수정 이력 audit_logs 기록) |
| 데이터 내보내기 | owner에게 CSV/Excel 내보내기 기능 제공 (법적 의무 이행 지원) |
| 서비스 해지 시 | 조직 해지 후에도 3년간 출퇴근 데이터 별도 보관, 데이터 인계 기능 제공 |

**주의:** 출첵 서비스가 종료되더라도 사업주의 보관 의무는 계속됨 → 서비스 종료 공지 시 데이터 내보내기 유예 기간 최소 90일 제공

### 4-2. 급여 명세서 교부 의무

**법적 근거:**
- 「근로기준법」 제48조 제2항 (2021.11.19 시행): 사용자는 임금을 지급할 때 임금명세서를 서면(전자문서 포함)으로 교부
- 위반 시: 과태료 30만원 (근로자 1인당)

**급여 명세서 필수 기재 항목:**

| # | 항목 | 출첵 구현 필드 |
|---|------|---------------|
| 1 | 성명 | profiles.name |
| 2 | 생년월일 | (추가 수집 필요 — 선택 항목으로) |
| 3 | 사업자등록번호/사업장명 | organizations.business_reg_number / name |
| 4 | 임금 지급일 | payroll_entries.paid_at |
| 5 | 근로일수 | 스케줄 기반 계산 |
| 6 | 총 근로시간 | payroll_entries.scheduled_minutes |
| 7 | 시간외/야간/휴일 근로시간 | (별도 계산 로직 필요) |

| 9 | 각 수당 항목 | (확장 필요 시) |

**구현 필요 사항:**
- 급여 명세서 화면 + PDF 다운로드 기능
- 직원에게 급여 확정 시 알림 발송 (notifications)
- 급여 명세서 데이터 3년 보관

**면책 고지:** 출첵은 급여 명세서 "생성 도구"를 제공하며, 법적 교부 의무는 사업주에게 있음을 이용약관에 명시

### 4-3. 최저임금법 준수 확인 로직

**법적 근거:**
- 「최저임금법」 제6조: 사용자는 최저임금액 이상의 임금을 지급해야 함
- 2026년 최저임금: 고용노동부 고시 기준 적용 (매년 갱신 필요)

**출첵 구현 방안:**

```
1. 최저임금 기준값 관리
   ├── 시스템 설정에 연도별 최저시급 저장
   ├── 매년 1월 자동 업데이트 (master 관리)
   └── 월 환산액 자동 계산

2. 사업주 시급 입력 시 검증
   ├── 입력 시급 < 최저시급 → 경고 표시
   │   "입력한 시급이 2026년 최저임금(XX,XXX원)보다 낮아요.
   │    최저임금 미만 지급은 법 위반이에요."
   └── 저장은 허용 (수습 감액 등 예외 가능)

3. 급여 계산 시 검증
   ├── 시급 × 근로시간 < 월 최저임금 환산액 → 경고
   └── 사업주에게 알림 발송
```

**주의:** 출첵은 "경고"만 표시하고 저장을 차단하지는 않음. 이유:
- 수습 기간(3개월) 최저임금 10% 감액 허용 (최저임금법 제5조)
- 감시/단속적 근로 등 예외 존재
- 최종 판단은 사업주 책임 → 이용약관에 면책 명시

### 4-4. 근로시간 관련 (주 52시간)

**법적 근거:**
- 「근로기준법」 제50조: 1주간 근로시간 40시간 초과 불가
- 제53조: 연장근로 12시간 한도 → 합계 주 52시간
- 5인 미만 사업장: 주 52시간 적용 제외 (연장근로 제한 없음)

**출첵 대상 사업장 특성:**
- F&B 소상공인 → 대부분 5인 미만 사업장
- 5인 미만은 주 52시간 미적용이나, 5인 이상 매장도 존재 가능

**출첵 구현 방안:**

```
1. 조직 설정에 "상시 근로자 수" 입력란
   ├── 5인 미만 / 5인 이상 선택
   └── 5인 이상 선택 시 주 52시간 모니터링 활성화

2. 주 52시간 모니터링 (5인 이상 매장)
   ├── 주간 근로시간 합계 자동 계산
   ├── 48시간 초과 시 owner에게 경고 알림
   ├── 52시간 초과 시 강조 경고
   │   "이번 주 [직원명]의 근로시간이 52시간을 초과했어요.
   │    5인 이상 사업장은 주 52시간을 넘길 수 없어요."
   └── 스케줄 편성 시 52시간 초과 예상 경고

3. 야간근로/휴일근로 표시
   ├── 22:00~06:00 근무 = 야간근로 표시
   ├── 주휴일 근무 = 휴일근로 표시
   └── 각각 가산수당 계산 참고 정보 제공
```

**면책:** 출첵은 근로시간 모니터링 도구이며, 근로기준법 준수 의무는 사업주에게 있음

```
   "크레딧 점수는 출첵 플랫폼 내 근태 이력을 수치화한 참고 지표이며,
    고용/해고/인사 등 법적 효력이 있는 판단의 근거가 아니에요.

```

## 6. 전자상거래법 (유료화 시)

### 6-1. 청약철회 (구독 서비스 특례)

**기본 원칙:**
- 소비자는 계약 체결일로부터 7일 이내 청약철회 가능 (법 제17조)
- 그러나 SaaS 구독 서비스의 경우 특례 적용 가능

**구독 서비스 청약철회 정리:**

| 상황 | 청약철회 | 이유 |
|------|---------|------|
| 구독 신청 후 7일 이내, 미사용 | 가능 (전액 환불) | 청약철회 기본 권리 |
| 구독 신청 후 7일 이내, 사용 중 | 가능 (이용 일수 공제) | 디지털 콘텐츠 특례 |
| 구독 7일 경과, 월간 구독 | 불가 (다음 갱신일 해지) | 계속적 서비스 |
| 연간 구독, 중도 해지 | 미사용 월 비례 환불 | 할인분 환수 가능 |

**구현 필요 사항:**
- 결제 즉시 "청약철회 안내" 메시지 발송
- 구독 관리 페이지에서 해지/환불 신청 가능
- 환불 처리 자동화 (PG사 API 연동)

### 6-2. 결제 수단별 규제

| 결제 수단 | 규제 사항 | 비고 |
|----------|----------|------|
| **신용카드** | 할부 항변권 안내 의무 (20만원 이상, 3개월 이상 할부) | 연간 구독 시 해당 가능 |

**권장 결제 수단 (SaaS 출시 시):**
1. 카드 결제 (PG사 — 토스페이먼츠, NHN KCP 등)
2. 카카오페이 간편결제
3. Apple/Google 인앱 결제 (앱스토어 배포 시)

### 6-3. 영수증 발급 의무

**필수:**
- 결제 완료 시 전자 영수증(거래 확인서) 발급

**영수증 포함 항목:**

- 환불/취소 안내

**세금계산서 발급:**

- 사업자 고객 요청 시 세금계산서 발급 의무 (부가가치세법 제32조)
- 전자세금계산서 발급 시스템 연동 필요 (홈택스 API 또는 세금계산서 SaaS)

## 8. 크레딧 시스템 법적 리스크

### 8-1. 전역 크레딧 = 일종의 평가 시스템 → 차별 소지?

**리스크 분석:**

출첵의 크레딧은 근태 데이터 기반 점수로, 사실상 **근로자 평가 시스템**에 해당할 수 있음.

| 리스크 유형 | 상세 | 심각도 |
|------------|------|--------|
| **채용 차별** | 사업주가 크레딧 점수 기반으로 채용/거절 → 「근로기준법」 제6조(균등처우) 위반 소지 | 중간 |
| **해고 사유** | 낮은 크레딧을 해고 사유로 사용 → 부당해고 분쟁 가능 | 중간 |
| **사생활 침해** | 이전 직장 근태 이력 노출 → 개인정보 자기결정권 침해 | 높음 |
| **알고리즘 차별** | 자동화된 점수 산정 → 「개인정보 보호법」 제37조의2(자동화된 결정에 대한 권리) | 높음 |

**「개인정보 보호법」 제37조의2 (2024년 시행):**
- 정보주체는 자동화된 결정에 대해 거부하거나 설명을 요구할 권리 있음
- 크레딧 점수 = 자동화된 의사결정에 해당 가능
- 대응: 점수 산정 기준 투명 공개, 이의 신청 기능 제공

### 8-2. 크레딧 공유 시 개인정보 이슈

**문제점:**

```
크레딧 카드 공유 = 다음 정보의 제3자 제공:
  - 크레딧 점수 (근태 평가)
  - 총 근무일수
  - 정시 출근율
  - 결근 횟수
  - 근무 이력 (어떤 조직에서 일했는지)

→ 이 모든 정보는 "개인정보"에 해당
→ 제3자에게 제공하려면 정보주체의 동의 필요
```

**그러나:**
- 크레딧 카드는 정보주체 **본인이** 공유를 선택 → 동의 문제 없음
- 다만, 사업주가 직원의 크레딧을 **강제로** 조회하는 것은 문제
  - owner가 소속 직원의 크레딧 조회: 서비스 목적상 허용 (이용약관에 명시)
  - owner가 비소속 사람의 크레딧 조회: **불허** (본인 공유만 가능)

### 8-3. 대응 방안

```
[방안 1] 법적 고지 명확화

  ① 크레딧 점수 산정 기준 전문 공개 (투명성)
     └── 마이페이지 > 크레딧 > "점수는 이렇게 계산돼요" 링크

  ② 크레딧 카드에 법적 면책 문구 삽입
     └── "이 점수는 출첵 플랫폼 내 근태 기록을 수치화한 참고 지표이며,
          고용/해고 등 법적 판단의 근거가 아니에요."

  ③ 이용약관에 크레딧 관련 조항 명시
     └── 점수 산정 방식, 이의 신청 절차, 면책 사항

[방안 2] 기능적 안전장치

  ① 크레딧 카드 공유 = 본인만 가능 (타인이 조회 불가)
  ② 크레딧 점수 비공개 옵션 제공
     └── 마이페이지 > 크레딧 > "내 크레딧 공개 설정"
     └── 비공개 설정 시 다른 조직 owner에게 미표시
  ③ 크레딧 이의 신청 기능
     └── "이 점수가 잘못됐다고 생각하면 이의를 신청해주세요"
     └── master 검토 후 조정 가능
  ④ 근무 이력 비공개 옵션
     └── 크레딧 카드에서 "근무 이력" 표시 여부 선택 가능

[방안 3] 자동화된 결정 대응 (개인정보 보호법 제37조의2)

  ① 자동화된 결정 거부권 안내
     └── 크레딧이 고용/인사에 활용되는 경우 거부 가능
  ② 크레딧 산정 로직 설명 요구권 보장
     └── 어떤 이벤트가 몇 점인지, 내 점수 변동 이력 조회 가능
  ③ 인적 개입 요구권
     └── 자동 산정 점수에 이의 시 사람(master)이 검토
```

## 9. 보험/배상

### 9-1. 서비스 장애로 인한 급여 오류 → 배상 책임?

**시나리오 분석:**

| 시나리오 | 책임 소재 | 리스크 |
|---------|----------|--------|
| 서비스 장애로 출퇴근 기록 누락 → 급여 과소 지급 | 출첵 일부 책임 가능 | 중간 |
| 계산 로직 버그로 급여 잘못 산출 | 출첵 책임 높음 | 높음 |
| 사업주가 시급/스케줄 잘못 입력 → 급여 오류 | 사업주 책임 | 낮음 |
| GPS 오차로 출퇴근 인정 불가 → 급여 미지급 | 기기/환경 문제, 출첵 책임 제한적 | 중간 |

**배상 책임 최소화 전략:**

```
1. 이용약관 면책 조항 (섹션 5-2)
   └── 급여 계산은 "참고용", 최종 확인은 사업주 책임

2. 급여 확정 프로세스
   ├── 자동 계산 → "초안" 상태
   ├── 사업주가 검토/수정 → "확정" 상태
   └── 확정 전까지 법적 효력 없음
   ★ 사업주의 "확인" 행위가 면책의 핵심 근거

3. 배상 한도 설정
   └── 이용약관에 배상액 상한 명시
       "회사의 배상 책임은 해당 월 서비스 이용료를 한도로 해요"
       (무료 기간에는 배상 의무 없음)

4. SLA (서비스 수준 계약)
   ├── 가용성 목표: 99.5% (월 3.6시간 미만 중단)
   ├── 목표 미달 시: 다음 월 이용료 비례 크레딧 제공
   └── 데이터 손실 시: 최근 백업으로 복원 (RPO 24시간)
```
